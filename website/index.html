Problem

Without Jarvis
  • Field techs read PRs on a 4-inch phone while wearing gloves.
  • "What can the assistant do?" returns a stale paragraph baked into the prompt.
  • Mid-conversation the server restarts; the user starts over.
  • The model guesses a fact and the tech trusts it because the voice is confident.
With Jarvis
  • Press to talk, hear an answer grounded in a live tool call.
  • "What can you do?" reads from the Tool Dispatcher's live capability list.
  • Server restart drops the WS; the client reconnects, the new session prompt carries the rolling summary plus last 20 turns.
  • Un-groundable claims become "I don't know" by prompt contract and by property test.

Components

Realtime Proxy src/proxy.ts
single Node process

One client WS in, one upstream WS out. Filters `jarvis.*` so they never reach the model. Renames the GA audio delta to the stable name for clients. Pre-flights `session.update` with the fully-nested audio shape.

Pros

One seam to instrument. Zero protocol drift visible to clients. Deterministic barge-in path under 300 ms.

Cons

One process per region; horizontal scale needs a sticky-session layer above it.
Tool Dispatcher src/tools/dispatcher.ts
zod schemas + structured errors

register / capabilities / openaiToolsSpec / dispatch. Validates args, gates by `available(env)`, wraps throws as `tool_failed` etc. The model never sees a stack trace.

Pros

Single enforcement point for the "no hallucination" contract. Capabilities are live, not hand-edited.

Cons

The hand-rolled zod→JSON Schema converter is intentionally narrow; new shapes need a deliberate extension.
Session Manager src/session.ts
pure function, 8-block prompt

Composes Identity + Capabilities + Memory + Preferences + Language + Hallucination guard + Slow-tool reminder + Barge-in reminder. Property test asserts the guard cannot be silently weakened.

Pros

Auditable. Order is fixed. Every block is collapsed to one line so token budget is predictable.

Cons

Prompt-level enforcement is necessary but not sufficient; that's why Slice 8 adds a deterministic post-call filter.
Tool surface src/tools/*
5 tools

wttr_get, memory_write, preference_set/clear, github_list_prs/list_issues/get_pr_comments/list_recent_merges/open_pr_for_issue. Slice 8 bubbles flagged authors to index 0 with `flagged: true`.

Pros

Each tool returns a JSON record with explicit error keys. Slow tools tag a filler the proxy speaks immediately.

Cons

Adding a new tool means a zod schema, a structured-error mapping, and updates to the property test if it changes the prompt surface.
SQLite memory src/db.ts
WAL, parameterised by user_id

4 tables: users, preferences, turns, memory_summaries. Every read query joins on user_id. Load test verifies 10 users see only their own rows.

Pros

Tests run against real DB files (no mocks). Trivial to back up — copy the file.

Cons

Single-writer. Move to Turso (`libsql`) for >~200 concurrent users per node.
Rolling Summarizer src/memory-summarizer.ts
gpt-4o-mini, 20-turn threshold

Fire-and-forget after every assistant turn. SELECTs the prior summary's turn_count; if `current - prior ≥ 20`, asks gpt-4o-mini to rewrite a 4-6 sentence summary. Failure preserves the prior summary.

Pros

Bounded cost; cheap. Never blocks the voice loop. Resilient to network failure.

Cons

Summary lag of up to 20 turns. Long sessions re-summarize multiple times.
Web client web/
Vite + AudioWorklet

Status pill, capability chip, 32-bar AnalyserNode visualizer, debug pane, mic-permission modal, `?demo=` + `window.__startDemo()` headless harness, 1 s × 3 WS retry.

Pros

Two AudioContexts (capture & playback) so format coercion is local; barge-in clears the queue in ~3 ms.

Cons

No service-worker offline mode; a flaky connection has the visible 3-retry budget and then surfaces.
iOS client ios/Jarvis
SwiftPM, SwiftUI, AVAudioEngine

Library + executable target. URLSessionWebSocketTask with 1 s × 3 retry. AVAudioConverter from hardware rate → 24 kHz Float32. Keychain stores userId (AfterFirstUnlock). Settings sheet flips server URL without a rebuild.

Pros

Type-augmentation on UserDefaults/Keychain reads idiomatic. Persists userId across reinstalls.

Cons

VoiceChat mode masks worse mic hardware; helmet-mic case wants a custom echo canceller.

Trade-offs

SQLite single-writer

Per-user inserts on the hot path.

Why I accept it

Per-user rows are tiny; WAL handles concurrent reads. Tests run against real DBs.

When it would bite

Beyond ~200 concurrent active users per node. Migration target is Turso (`libsql`).

In-memory weather cache

60 s TTL, per-process.

Why I accept it

wttr.in is slow; the cache saves the user the latency on a repeated ask.

When it would bite

Multi-region deploys see TTL skew. Move to a per-region Redis if needed.

No upstream silent reconnect

If OpenAI drops, the proxy closes the client; the client retries.

Why I accept it

Avoids partial-state sessions; the new session has a fresh prompt.

When it would bite

A user in a moving vehicle would prefer invisible reconnect.

Hand-rolled zod→JSON Schema

Covers our shapes only.

Why I accept it

Fewer dependencies, smaller audit surface.

When it would bite

A new tool needing unions or refinements requires a deliberate extension.
